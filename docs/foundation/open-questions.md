# Foundation Release open questions

Resolve release-blocking questions before implementation or release, as indicated.

| ID | Question | Owner | Needed by | Blocking | Status |
| --- | --- | --- | --- | --- | --- |
| OQ-001 | Who is the primary Foundation Release user? | Product | Scope approval | Yes | Open |
| OQ-002 | What exact job and primary workflow must the Foundation Release prove? | Product | Scope approval | Yes | Open |
| OQ-003 | What is the primary domain record called, and what fields are required? | Product/Engineering | Data model | Yes | Open |
| OQ-004 | Is data owned by a user, workspace, or organization? | Product/Engineering | Architecture | Yes | Open |
| OQ-005 | Which authentication model will the Foundation Release use? | Engineering/Security | Architecture | Yes | Open |
| OQ-006 | What technology stack and hosting platform are approved? | Engineering | Implementation start | Yes | Open |
| OQ-007 | Does the Foundation Release handle personal, financial, or regulated data? | Product/Security | Security review | Yes | Open |
| OQ-008 | What are the record retention and deletion rules? | Product/Security | Data model | Yes | Open |
| OQ-009 | What measurable outcome defines Foundation Release success? | Product | Release | Yes | Open |
| OQ-010 | Who owns production operation and incident response? | Engineering | Release | Yes | Open |
| OQ-011 | What performance and availability targets apply? | Product/Engineering | Release | No | Open |
| OQ-012 | What accessibility target applies? | Product/Design | UI completion | Yes | Open |

## Resolution process

1. Assign one accountable owner.
2. Record the answer and decision date.
3. Create an ADR when the answer changes architecture or a cross-cutting product rule.
4. Update every affected Foundation Release document.
5. Mark the question resolved and link the final source of truth.
